# Utils Layer

## File responsibilities

- **idbSchema.ts** — Single source of truth for the `id_card_store` IndexedDB database: name, version, and the object store names (`workspaceList`, `workspaceData`, `userTemplates`, `printPresets`, `meta`). All consuming modules import `STORE_NAMES`/`ALL_STORE_NAMES` from here rather than passing literal strings, so whichever module opens the connection first still creates every store in one `onupgradeneeded` pass.
- **idbStore.ts** — Generic async key/value table helper (`createIdbTable<T>(storeName)`) shared by `workspaceStorage.ts`, `userTemplates.ts`, and `printPresets.ts`: `get`, `getAll`, `getAllKeys`, `put`, `putMany`, `delete`, `clear`. Every method resolves to a safe default (`null` / `[]` / `false`) instead of throwing when IndexedDB is unavailable (private browsing, disabled, etc.) — callers that need a real fallback check the boolean/null return.
- **storageMigration.ts** — One-time migration of structural app data from the pre-v2 localStorage keys into the `id_card_store` IndexedDB database. `runMigrationIfNeeded()` is idempotent (write-once beacon in localStorage) and non-destructive (`PURGE_LEGACY_KEYS = false` — legacy keys are left in place during the soak period; verified only after read-back). `getMigrationNoticeIfAny()` reports any corrupt/unverifiable entries exactly once. Must be awaited (in `App.tsx`'s boot effect) **before** any read from `workspaceStorage.ts`/`userTemplates.ts`/`printPresets.ts`, since those now read from IndexedDB exclusively. Every localStorage access goes through internal guarded `ls*` helpers (get/set/remove/keys, each `try/catch`'d) so a storage-locked browser (private mode / disabled by policy) degrades to "no legacy data" instead of throwing out of the boot path — the idb `meta` record still short-circuits re-copying when the beacon can't persist, so idempotency holds. `readLegacyWorkspaceList()` / `readLegacyWorkspaceData(id)` are **synchronous read-only** fallbacks used by `App.tsx` only when IndexedDB is unavailable, so an existing user keeps read access to their workspaces (writes still fail, surfaced via the storage-full banner); they intentionally skip the copy-on-write parent-template overlay.
- **workspaceStorage.ts** — Async IndexedDB CRUD (via `idbStore.ts`) for the workspace list and per-workspace project data; defines `WorkspaceMeta`, `WorkspaceData`, `WorkspaceListState`. Every exported function (`getWorkspaceList`, `saveWorkspaceData`, `createSubWorkspace`, `deleteWorkspaceTree`, `duplicateWorkspace`, etc.) returns a `Promise` — there is no synchronous read path. `LIST_KEY`/`DATA_PREFIX` are kept exported only as the legacy localStorage key names `storageMigration.ts` reads from.
- **assetStore.ts** — content-addressed IndexedDB store (db `id_card_assets`, a separate database from `id_card_store`) for large data URLs. `externalizeWorkspaceAssets` (sync) swaps template background/watermark images and card photo overrides above 8KB for `asset:<hash>` refs at persistence time; `resolveWorkspaceAssets` (async) swaps them back at load time. See "Asset store" section below.
- **workspaceFile.ts** — File System Access API save/open for `.idcard` workspace files and `.idtemplate` template files; also owns FSA type declarations and autosave-pref helpers.
- **fileHandleStore.ts** — IndexedDB wrapper (`setStoredHandle`, `getStoredHandle`, `getAllStoredHandles`, `deleteStoredHandle`) that persists `WorkspaceFileHandle` objects keyed by root workspace id, so a workspace's `.idcard` file link survives a page reload. Every function resolves to a safe default (`null` / `void` / empty `Map`) instead of throwing — private browsing / IndexedDB-disabled degrades silently to the pre-existing in-memory-only behavior. This remains its own database (`id_card_file_handles`), separate from `id_card_store`.
- **userTemplates.ts** — Async IndexedDB CRUD (via `idbStore.ts`) for user-saved templates (list of `{ meta, template }` entries). Stored templates hold `asset:` refs for large images (externalized on save); `loadUserTemplates` returns them unresolved (fine for list rendering), `loadResolvedUserTemplates` / `resolveTemplateAssets` resolve back to data URLs. `saveUserTemplate`, `deleteUserTemplate`, and `restoreUserTemplates` return `Promise<boolean>` — `false` on failure; callers must surface it. `restoreUserTemplates` replaces the whole stored set (`table.clear()` then `putMany`).
- **csv.ts** — Thin PapaParse wrapper; parses a `File` into `{ headers, rows }`.
- **exportImages.ts** — Renders cards off-screen via `CardCanvas`, captures each with html2canvas. `renderCardsToImages` is the shared rendering core (returns `{ recordIndex, dataUrl, blob }[]`); `exportCardsAsImages` calls it, bundles into a ZIP (with a `manifest.json` recording card mm size + format), and saves via `saveBlob`. `exportCardImagesToZip(cards, format, fileName, onProgress?)` is a second, narrower zip helper for `CombinePdfDialog.tsx`'s "Images (ZIP)" output mode — it takes already-rendered `CardImage[]` (the same shape `aggregateCardsToPdf` consumes, from `aggregatePdf.ts`) rather than a template/records, converts each `dataUrl` to a `Blob` via `fetch()`, and zips them **without** a manifest (combined sources can mix card sizes, so a single-size `ExportManifest` doesn't apply).
- **aggregatePdf.ts** — Shared jsPDF engine. `aggregateCardsToPdf(cards, opts)` packs `CardImage[]` (data URL + mm size) densely into one PDF, grouping by exact mm size so mixed sizes each get their own page run. Reuses `computeLayout`/`computeEffectivePaperDims` from `PrintSettings.tsx`. This is what eliminates per-workspace last-page waste.
- **importImages.ts** — `importCardsFromFiles(files)` reads picked ZIPs/images into `{ sized, unsized, warnings }`. ZIPs with a `manifest.json` come back fully sized; loose images / manifest-less ZIPs come back `unsized` and the caller must supply dimensions.
- **saveFile.ts** — `saveBlob(blob, fileName, accept)` (FSA save picker with `<a download>` fallback) and `safeFileName(name)`. Used by `exportImages.ts` and `aggregatePdf.ts`.
- **backup.ts** — Full-app backup (all workspaces + user templates + print presets) to a JSON download; and restore from that JSON back into the IndexedDB-backed storage modules. `createBackup`/`restoreFromBackup` are both fully async, awaiting `getWorkspaceList`/`getWorkspaceData`/`saveWorkspaceData`/`saveWorkspaceList`/`restoreUserTemplates`/`replacePrintPresets`.
- **file.ts** — Single helper: `readFileAsDataUrl(file)` — reads a `File` as a base64 data URL.
- **testHelpers.ts** — Shared Vitest helpers. `clearAllStores()` wipes every `id_card_store` object store between tests; the storage-layer suites call it in `beforeEach` (see "Tests" below).
- **id.ts** — `generateId(prefix?)` — monotonic unique element IDs (prefix + timestamp + counter).
- **printPresets.ts** — Async IndexedDB CRUD (via `idbStore.ts`) for user-saved print presets. `loadPrintPresets`/`savePrintPreset`/`deletePrintPreset` are `Promise`-returning; `replacePrintPresets` (clear + `putMany`) backs `backup.ts`'s restore path.
- **a11y.ts** — `touchTarget44`, an `sx` snippet (`position: relative` + an absolutely-centered, invisible 44×44 `::after`) that expands a control's tappable area to the WCAG/Material 44px minimum without inflating its visible size. Spread onto any MUI `Checkbox`/`IconButton` that renders smaller than 44px (e.g. `size="small"`); requires the target to keep normal position layout (not `position: fixed`).

## Persisted storage

All structural app data (workspace list, per-workspace project data, user templates, print presets) lives in the **`id_card_store`** IndexedDB database (see `idbSchema.ts` for the store names). Large images live in the separate `id_card_assets` database (see "Asset store" below), and persisted FSA file handles live in the separate `id_card_file_handles` database (see `fileHandleStore.ts`).

| IndexedDB store (`id_card_store`) | Key | What lives there |
|-----|-----|-----------------|
| `workspaceList` | `'current'` | `WorkspaceListState` — `{ currentId, workspaces: WorkspaceMeta[] }` |
| `workspaceData` | workspace id | `WorkspaceData` for each workspace (template, records, columnMapping, printPresets, printSettings, selectedCardIndices, currentTemplateSource, logo) |
| `userTemplates` | template id | `{ meta: UserTemplateMeta; template: Template }` |
| `printPresets` | preset id | `PrintPreset` |
| `meta` | `'migration_v1'` | Migration status/results — see `storageMigration.ts` |

`id_card_autosave_to_file` (`'true'` / `'false'`, autosave-to-file preference, default on) remains a plain localStorage key — it's a small UI preference, not structural data, and was never part of this migration.

**Legacy localStorage keys** (`id_card_workspace_list`, `id_card_workspace_data_{id}`, `id-card-user-templates`, `id-card-print-presets`) are read only by `storageMigration.ts`, once, on first boot after upgrade. They are **not purged** (`PURGE_LEGACY_KEYS = false` during the soak period) — do not rely on their absence to mean "no legacy data", and do not read them directly from anywhere except `storageMigration.ts`.

`csvData` is written as part of each workspace's `WorkspaceData` (via `saveWorkspaceData`) so column-mapping survives page reload. It is stripped from `.idcard` file exports (`buildFileContent` destructures it out) and from workspace duplicates (`duplicateWorkspace` strips it).

`saveWorkspaceData` side-effects: if `data.logo` is set, it awaits `updateWorkspaceMeta` to keep the list entry's logo in sync.

## Copy-on-write template inheritance (`templateLinkedToParent`)

Sub-workspaces are created with `templateLinkedToParent: true` in their `WorkspaceData`: their stored `template` is only a fallback snapshot, and the *effective* template is the parent's current one. **Load stored data through `getEffectiveWorkspaceData(id)`, not `getWorkspaceData(id)`,** whenever the data feeds app state or rendering (workspace switch, hydration, delete-then-load, Combine-PDF). It overlays the parent's template when linked, and self-heals to a detached copy if the parent record is missing. Detachment happens in the reducer: any template mutation clears the flag in `AppState`, and the next save persists the workspace's own snapshot. `duplicateWorkspace` duplicates from *effective* data and strips the flag (a duplicate under a different parent must not adopt that parent's design). The flag round-trips through `.idcard` files and backups (`buildFileContent` keeps it). Raw `getWorkspaceData` remains correct for save-to-file paths, which intentionally write the snapshot + flag.

## Asset store — large data URLs never go in the structural store

The old localStorage ~5MB quota is why this layer exists (a template background image is a multi-MB base64 data URL, and every sub-workspace/duplicate used to get a full copy, silently exhausting the quota — workspaces then lost their template on reload, the original bug). It still exists post-migration: even in IndexedDB, keeping every workspace's full-size images inline would balloon `workspaceData` records and duplicate large blobs across sub-workspaces/copies for no reason — `assetStore.ts`'s content-addressed dedup is the actual fix, independent of which database backs the structural data.

- **Write path**: `saveWorkspaceData` calls `externalizeWorkspaceAssets` before persisting. Data URLs > 8KB in `template.background.value`, `template.watermark.value`, and `record.overrides` values are replaced with `asset:<fnv1a-hash>-<len>` refs; the blobs go to the `id_card_assets` IndexedDB database (fire-and-forget, deduped by content hash + in-memory `persisted` set). The `workspaceData` record stays small. `saveWorkspaceData` returns **`Promise<boolean>`** — `false` on failure. Creation flows (`handleNewSubWorkspaceConfirm`) must await the write first, check the result, and abort registration on failure.
- **Read path**: `getWorkspaceData` still returns raw stored data, which may contain `asset:` refs. **Every consumer that feeds data into app state or a self-contained artifact must `await resolveWorkspaceAssets(data)` first.** Current resolve sites: App.tsx hydration + autosave-to-file, WorkspaceSwitcher (`doSwitch`, sub-workspace creation, delete, save-to-file, both duplicate flows), CombinePdfDialog `generateFromWorkspaces`, backup.ts `createBackup`. In-memory `AppState` and all rendering code only ever see plain data URLs.
- **Self-contained artifacts**: `.idcard` files and backup JSON must contain real data URLs (portable across machines), so resolve before `writeWorkspaceToHandle`/`buildFileContent`; `restoreFromBackup`/`restoreWorkspaceFile` re-externalize automatically by routing through `saveWorkspaceData`.
- **Migration-free**: pre-existing stored data with inline data URLs passes through `resolveWorkspaceAssets` untouched and is externalized on its next save.
- **Missing asset** (e.g. IndexedDB cleared): `resolveWorkspaceAssets` drops the background/watermark to `null` (and clears a missing card-override photo to `null`), each with a console.warn, rather than rendering a broken `asset:` string.
- **User templates** (`userTemplates` IndexedDB store) follow the same pattern via the exported template-level helpers `externalizeTemplateAssets` / `resolveTemplateAssets`: `saveUserTemplate` externalizes on write; `TemplatePicker.handleSelectUser` resolves before `onSelect` (so `DesignStep.handleSelectTemplate` and app state only see data URLs); `saveTemplateWithPicker` resolves before writing `.idtemplate` files; backup export uses `loadResolvedUserTemplates` and restore routes through `restoreUserTemplates`.

## FSA types — do not add @types/wicg-file-system-access

`lib.dom` does not include File System Access API types. Local declarations (`FSWritable`, `WorkspaceFileHandle`, `FSAFileHandle`, `SavePickerOpts`, `OpenPickerOpts`, `WindowWithFSA`, `FSAPermissionState`) live at the top of `workspaceFile.ts`. Do not install `@types/wicg-file-system-access` — use the local types.

`WorkspaceFileHandle` includes a `name: string` field (the filename) so the UI can display the linked file name. When constructing a mock or stub handle, always include `name`.

`WorkspaceFileHandle` also declares `queryPermission?`, `requestPermission?`, and `isSameEntry?` as **optional** methods — real FSA handles always have them, but hand-built mock/stub handles or older browsers may not. Every call site must feature-detect with `typeof handle.xxx === 'function'` before calling, the same pattern as `hasSaveFilePicker()`/`hasOpenFilePicker()`. Do not make these required — that would force every mock handle in the codebase to fake them.

## IndexedDB — persisted file handles (fileHandleStore.ts)

`FileSystemFileHandle` objects are natively structured-cloneable, so `fileHandleStore.ts` stores them directly in IndexedDB (database `id_card_file_handles`, object store `handles`, keyed by root workspace id) — no serialization needed. This is what lets a workspace's `.idcard` link survive a page reload; before this, `fileHandleRef` in `App.tsx` was purely in-memory and every reload silently dropped the link. See the "WorkspaceSwitcher — file handle rehydration and reconnect" gotcha in `src/components/CLAUDE.md` for how this is wired into the UI.

Note: plain mock handle objects with function properties (e.g. in manual browser testing) are **not** structured-cloneable — only real `FileSystemFileHandle` instances have the browser's native serialization support. `setStoredHandle` catches this via the same "resolve to a safe default" contract, so a non-cloneable value fails silently rather than throwing.

`exportImages.ts` has its own inline FSA type aliases (`FSWritable`, `FSFileHandle`, `SavePickerOpts`) scoped to that function. This is intentional — they only cover `BufferSource | Blob | string` writes, not the workspace file write shape.

## FSA fallback — always provide one

FSA only works in Chrome/Edge. Every FSA call must be guarded and fall back gracefully:

- **Save workspace** — `saveWorkspaceWithPicker` falls back to `downloadWorkspaceFile` when `showSaveFilePicker` is absent. `pickSaveFileHandle` splits the picker-acquisition step out on its own (no write) — callers with slow data to gather before writing (e.g. `WorkspaceSwitcher.handleSaveWorkspace`, resolving assets across a whole workspace tree) must call `pickSaveFileHandle` *first* and gather data after, since `showSaveFilePicker`'s required transient user activation can expire if the picker is requested only once all that data is ready. `saveWorkspaceWithPicker` itself still bundles picker+write for callers whose data is already in hand (e.g. `handleNewWorkspaceConfirm`, saving a fresh empty workspace).
- **Open workspace** — `openWorkspaceWithPicker` / `openWorkspaceFilePickerWithHandle` return `null` when `showOpenFilePicker` is absent; callers must use a hidden `<input type="file">` as the fallback path.
- **Save template** — `saveTemplateWithPicker` falls back to an `<a download>` click.
- **Export ZIP** — `exportCardsAsImages` falls back to `URL.createObjectURL` + `<a download>` click.

Use `hasSaveFilePicker()` / `hasOpenFilePicker()` (exported from `workspaceFile.ts`) to check availability before calling FSA functions.

## html2canvas background rule — use `<img>`, not `background-image` CSS

html2canvas ignores the `scale` parameter when rendering CSS `background-image` properties. The upscaled image will be blurry/pixelated. `CardCanvas` renders image backgrounds as `<img>` elements so html2canvas captures them correctly via `drawImage()`. Do NOT switch image backgrounds back to CSS `background-image`.

`exportImages.ts` deliberately avoids CSS transforms on the wrapper div for the same reason — all upscaling is folded into html2canvas's own `scale` parameter (`h2cScale`).

`waitForContainerReady` in `exportImages.ts` waits for `document.fonts.ready` and for all `<img>` elements in the container to load before calling html2canvas. If you add new async visual resources to `CardCanvas`, they must be awaitable here too.

## User template sync when importing .idcard files

`LOAD_WORKSPACE_STATE` (Redux action) does NOT auto-sync user templates embedded in workspace data into the `userTemplates` IndexedDB store. When importing an `.idcard` file in `WorkspaceSwitcher.tsx` (or anywhere), user templates present in the workspace data must be explicitly persisted by `await`ing `saveUserTemplate` for each one. Skipping this causes the imported templates to be invisible in the template picker.

## csv.ts quirks

- `FieldMismatch` parse errors are silently filtered out — only fatal errors reject the promise.
- Column headers come from `results.meta.fields` (PapaParse's canonical source); `Object.keys(rows[0])` is only a fallback for edge cases where PapaParse doesn't populate `meta.fields`.
- Parsed CSV rows live only in `AppState.csvData` at parse time. That state field is itself persisted as part of each workspace's `WorkspaceData` (see `workspaceStorage.ts` / `src/store/CLAUDE.md`), so it does survive reload/workspace-switch — `csv.ts` itself has no persistence of its own; parsing happens fresh from the uploaded `File` every time.

## Workspace hierarchy

Sub-workspaces are max one level deep (enforced in `createSubWorkspace`). `WorkspaceMeta.parentId` marks children. Always use `await deleteWorkspaceTree(id)` to delete a workspace — it removes the entry and all its direct children atomically.

## getWorkspaceList — empty fallback on first launch

`getWorkspaceList()` resolves to `{ currentId: '', workspaces: [] }` when the `workspaceList` IndexedDB store has no `'current'` record, or when the stored list has no workspaces. It does **not** synthesise a ghost `{ id: 'default', name: 'Default' }` entry. All callers must handle an empty `workspaces` array. The app shows the first-launch setup modal when the list is empty — `App.tsx`'s boot effect awaits `runMigrationIfNeeded()` then `getWorkspaceList()` and sets `needsSetup` if `workspaces.length === 0` (there's no synchronous localStorage check anymore, since the source of truth is IndexedDB). The whole UI is gated behind a `bootResolved` state — a `CircularProgress` renders until the boot effect settles (in a `try/finally`, so the spinner always clears) — so neither the app nor the setup modal flashes before the list is known. When `runMigrationIfNeeded()` reports `degraded` (IndexedDB unavailable), the boot effect falls back to `readLegacyWorkspaceList()`/`readLegacyWorkspaceData()` instead of returning early, so an existing user isn't left on a blank app with no modal. `deleteWorkspaceTree` still writes a `'default'` fallback when the last workspace is deleted (pre-existing behaviour, separate concern).

## Tests (Vitest + fake-indexeddb)

`idbStore.test.ts`, `storageMigration.test.ts`, `workspaceStorage.test.ts`, `userTemplates.test.ts`, `printPresets.test.ts`, `backup.test.ts`, `assetStore.test.ts`, `fileHandleStore.test.ts`, `workspaceFile.test.ts`, and `exportImages.test.ts` sit alongside the modules they test and run under Vitest with a jsdom environment (`vitest.config.ts` at the repo root, `fake-indexeddb/auto` as a setup file so `indexedDB` exists in Node; `src/setupTests.ts` polyfills several other jsdom gaps — see below). Run with `npm run test`. `*.test.ts`/`*.test.tsx` files are excluded from `tsconfig.app.json`'s `include` so they never affect `npm run build`'s `tsc -b` pass.

`idbStore.test.ts` and `storageMigration.test.ts` cover the degraded-mode contract (every op resolves to a safe default when `indexedDB` is deleted from `globalThis`, using `vi.resetModules()` + a fresh dynamic `import()` so the module's cached connection promise doesn't leak between tests) — this is the only environment that can actually simulate "IndexedDB absent before any module runs"; a real browser reload always restores the native `indexedDB` global, so this scenario can't be reproduced by manually toggling it in a live page. `storageMigration.test.ts` also covers the synchronous `readLegacyWorkspaceList`/`readLegacyWorkspaceData` fallbacks (normal read, `currentId` defaulting, and corrupt/absent → `null` without throwing). `fileHandleStore.test.ts` has the same degraded-mode coverage for its own separate `id_card_file_handles` database.

`workspaceStorage.test.ts`, `userTemplates.test.ts`, `printPresets.test.ts`, and `backup.test.ts` cover the storage-layer user flows: workspace create/rename/switch/duplicate(root+sub)/delete, sub-workspace copy-on-write template inheritance (including the self-heal-when-parent-missing edge case), user-template and print-preset CRUD plus their "replace all" backup-restore semantics, and a full `createBackup` → wipe → `restoreFromBackup` round-trip. Every test suite calls the shared `clearAllStores()` helper (exported from `testHelpers.ts`: `Promise.all(ALL_STORE_NAMES.map((name) => createIdbTable(name).clear()))`) in `beforeEach` so the tests stay isolated — reuse that helper rather than deleting the whole IndexedDB database, since `fake-indexeddb`/real IndexedDB can block or race a `deleteDatabase` call while a connection from an earlier test is still open.

`assetStore.test.ts` covers content-addressed dedup, the in-memory-cache-then-IndexedDB resolve order (using `vi.resetModules()` per test — see below — so `memCache`/`persisted` don't leak between tests, then verifies a value written by one module instance is still resolvable by a freshly-imported instance sharing the same underlying fake-indexeddb database), the inline-vs-externalize 8KB threshold, and the missing-asset-drops-to-null-with-warning self-heal path. Because `assetStore.ts` keeps module-level singleton state (`memCache`, `persisted`) rather than going through `idbStore.ts`, every test calls `vi.resetModules()` + dynamic `import('./assetStore')` in a `beforeEach`/per-test helper, unlike the other storage-layer suites which just call `clearAllStores()`.

`fileHandleStore.test.ts` covers put/get/delete/getAll round-trips, including the case where `put()` throws `DataCloneError` *synchronously* for a non-structured-cloneable value (e.g. a hand-built mock handle with function properties) rather than routing through `tx.onerror` — `setStoredHandle` wraps that call in a try/catch (mirroring `openDb()`) so it resolves to a safe default instead of throwing, matching the module's documented contract.

`workspaceFile.test.ts` mocks `window.showSaveFilePicker`/`showOpenFilePicker` per test (deleted in `afterEach`) to cover both the FSA path and the non-FSA download fallback for save/open, `AbortError` (cancel) vs. other-error handling, `csvData` stripping from both the root and child workspaces before write, and the `isWorkspaceFile`/`isTemplateFile` format validators.

`exportImages.test.ts` mocks the `html2canvas` and `./saveFile` modules (`vi.mock`) so `renderCardsToImages`/`exportCardsAsImages` can run against a real `CardCanvas` render without needing actual canvas rasterization; covers per-card error isolation (one failing card doesn't abort the batch), `onProgress` sequencing, off-screen container cleanup, PNG-vs-JPEG mime/quality pass-through to `canvas.toBlob`/`toDataURL`, and the generated ZIP's `manifest.json` contents (verified by re-reading the saved blob back through `JSZip`).

### jsdom gaps polyfilled in `src/setupTests.ts`

jsdom (as of the version pinned here) doesn't implement several browser APIs this codebase depends on. Rather than mocking these per-test, they're polyfilled once, globally, so product code runs under test exactly as it does in a real browser:

- `Blob.prototype.text()` / `.arrayBuffer()` — missing entirely; polyfilled via `FileReader` (which jsdom does implement). Needed by `workspaceFile.ts`'s `readWorkspaceFile`/`readTemplateFile` (`file.text()`).
- `URL.createObjectURL` / `revokeObjectURL` — missing entirely; stubbed to a fixed string / no-op so they exist for `vi.spyOn()` to override per-test (`spyOn` requires the property to already exist).
- `document.fonts` — the CSS Font Loading API isn't implemented; stubbed with `{ ready: Promise.resolve() }` so `exportImages.ts`'s `waitForContainerReady` doesn't hang.
- `ResizeObserver` — missing entirely; stubbed as a no-op class. Needed by `CardCanvas`'s `FitText` re-measurement effect, so this matters for *any* test that renders `CardCanvas` (directly or via `exportImages.ts`), not just canvas/file-specific ones.

If a new test hits a "not implemented" jsdom error or a `ReferenceError` for a browser global, check whether it belongs here (a true environment gap other tests will also hit) before reaching for a per-test workaround.

`src/setupTests.ts` also registers a global `afterEach(() => cleanup())` (from `@testing-library/react`). This project imports `describe`/`it`/`expect`/etc. explicitly from `'vitest'` rather than enabling `test.globals: true`, and RTL's own auto-cleanup only self-registers when it detects a *global* `afterEach` — without this, unmounted trees (and anything rendered into a portal, e.g. MUI `Dialog`/`Select`/`Menu`) leak into the next test in the same file, causing "found multiple elements" failures that have nothing to do with the component under test. Every component test file relies on this; do not add a competing manual `afterEach(cleanup)` per file.

For MUI `Select`/`Autocomplete` components in tests: query by `getByRole('combobox', { name })` when the component links its label via `labelId`/`id`; when it doesn't (see `ColumnMapping.tsx`'s flagged gap in `src/components/CLAUDE.md`), fall back to `getAllByRole('combobox')` and index by render order.

### `src/testUtils.tsx` — `renderWithAppState`

`AppStateProvider` (`src/store/AppStateContext.tsx`) hardcodes `useReducer(appReducer, initialState)` with no prop to inject a custom starting state, which every step/sub-component that reads `useAppState()`/`useAppDispatch()` needs for anything beyond the default empty state. `renderWithAppState(ui, { initialState })` wraps `ui` in a real `AppStateProvider` and, when `initialState` is given, seeds it via dispatches fired from a child component's `useEffect` on mount — dispatching during render itself would update the *ancestor* `AppStateProvider`'s state mid-render, which React only supports for a component adjusting its own state, not another component's. Because seeding happens in an effect (post-mount, before paint is visible to RTL), tests that depend on seeded content must use `find*` queries or `waitFor`, not a synchronous `getBy*` right after `renderWithAppState`. This file is excluded from `tsconfig.app.json` (alongside the `*.test.ts(x)` patterns) so it never affects `npm run build`, even though its `.tsx` extension doesn't match those globs on its own.

`initialState` accepts any `Partial<WorkspaceData>` **plus** `currentWorkspaceId`/`workspaceList`. The latter two are on `AppState` but are *not* part of `WorkspaceData` (they're workspace-switcher UI state set via their own `SET_CURRENT_WORKSPACE`/`SET_WORKSPACE_LIST` actions, not persisted per-workspace) — `LOAD_WORKSPACE_STATE` silently ignores them if included in its payload. `Seed` splits them out and dispatches the two extra actions separately. This surfaced while testing `PrintStep.tsx` (which reads `workspaceList`/`currentWorkspaceId` to derive the export file name) — if you add a new field to a step component that reads something outside `WorkspaceData`, check whether `Seed` needs to grow another special case rather than assuming `LOAD_WORKSPACE_STATE` covers it.
